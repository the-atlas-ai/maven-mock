Atlas · Design Review
Project Orchestration Layer
Unified model for client engagements — automated, manual, and hybrid work under one roof
Backend · Django Frontend · Next.js Status: Pre-implementation Author: Engineering · Apr 2026
01
Problem
CPA firms run multi-step client engagements. Today, work is either fully automated (workflows) or fully manual (ad hoc). There's no unified view of what's in flight, what's blocked, or what a person should work on next.
🗂️
No engagement container
Automated workflow runs exist in isolation. There's no "Tax Prep 2025 — Mesa Verde" record that owns all the work for that engagement.
🔀
Mixed work has no home
An engagement includes automated steps (import data) and human steps (CPA review, client sign-off). These can't coexist in the current workflow model.
🧑‍💼
No person-facing queue
A CPA has no single view of "what do I need to do right now across all my clients." Work is discovered by checking each workflow run individually.
02
Mental Model
Three layers. Each has a distinct job. None duplicates the other.
Project
Client Period Owner Deadline Status Metadata
User-facing anchor — who, what, when. CPA thinks in projects, not workflows.
↓  1 : many
Tasks
Template-defined Workflow-emitted Manual Automated parent_task depends_on
Work items. Coarse-grained milestones from template + fine-grained items emitted at runtime by workflows.
↓  automated tasks trigger
Workflow Engine
WorkflowExecution Nodes Internal DAG WorkflowTrigger
Automation plumbing. Invisible to the CPA. Handles parallelism, node sequencing, and emitting child tasks when human review is needed.
Project is a thin identity layer. Workflow engine is the execution layer. They do not duplicate each other.

04
Task Taxonomy
Every task is one of two types. Source (where it came from) is orthogonal to type (how it executes).
⚡ Automated Task auto
Executes byTriggering a workflow. System runs it without human input.
Starts whenAll depends_on tasks complete. System triggers immediately.
Completes whenWorkflow done AND all child tasks (emitted) are done.
Can emitChild manual tasks at runtime (e.g. "Review 12 flagged items").
ExampleImport Bank Data, Categorize Transactions, E-file Return
👤 Manual Task manual
Executes byA human. Assigned to a person by role (CPA / Operator / Client).
Starts whenAll depends_on complete. Appears in assignee's dashboard queue.
Completes whenHuman marks it done. Optional: attach output document.
Can emitNothing. Manual tasks are leaf nodes.
ExampleGather Documents, CPA Review, Client Approval, Sign-off

💡
Source is separate from Type
A task can be template-defined (created at project instantiation) or workflow-emitted (created at runtime by a workflow node). Either can be manual type. Automated tasks are always template-defined — a workflow can't emit another automated task, only a manual review task.
05
Task Creation Paths
Tasks enter the system through two distinct paths. Both are first-class.
📋  Path A — Template Instantiation
1
CPA creates new project — selects template type (Tax Prep / Book Close), client, period, deadline.
↓
2
System loads ProjectTemplate and iterates its TaskDefinitions.
↓
3
One Task record is created per definition. All start as todo. Depends_on, type, role copied from definition.
↓
4
Tasks with empty depends_on become ready immediately. Automated ones trigger their workflow. Manual ones notify the assignee.
⚡  Path B — Workflow Emission
1
An automated project task triggers a workflow. Workflow runs its internal node sequence.
↓
2
A node reaches a checkpoint (e.g. cluster_review finds flagged items). Node decides human review is needed.
↓
3
Node emits a child Task via post_save signal. Task gets parent_task = triggering project task, source = emitted, type = manual.
↓
4
Child task appears in assignee's queue. Parent task stays in_progress until child is done AND workflow completes.

11
Open Decisions — For Review
Items that need a call before implementation starts.
Decision Options Status
Template storage
Where do template definitions live?		 A) Code + seed command (simpler, version-controlled, not user-editable)  ·  B) Fully DB-backed (firms can customize templates, more complex). Recommend A for v1, migrate to B later. Open
DAG complexity
How far does depends_on go in v1?		 V1 is linear (each task depends on previous). The depends_on JSON array already supports parallel branches — upgrading is a workflow engine change, not a model change. No action needed now. Decided
Client tasks
assignee_role = client — how does client receive and complete tasks?		 A) Email link → simple web form to mark complete  ·  B) Client portal (future). For v1, recommend email link with a token-authenticated approval endpoint. Open
Automated task for future workflows
"Compute Return" and "File Return" don't have workflows yet.		 Template tasks with workflow_key = null behave as manual tasks as a fallback. CPA manually marks them done until the workflow exists. No blocking dependency. Decided
Parallel task unlock
Can multiple tasks start at the same time?		 Yes — any task whose full depends_on set is satisfied unlocks simultaneously. Orchestrator loops all candidates after each task completion. No model change required. Decided
Project visibility
Can all team members see all projects?		 Today: team-scoped (all members see all projects). Future: per-project ACL. Out of scope for v1 — use existing TeamContextModelMixin. Decided
